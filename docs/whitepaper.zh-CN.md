# n8n Workflow Builder 白皮书

## 摘要

`n8n-workflow-builder` 是一个面向 Codex 的专用技能，目标是把用户的自然语言流程描述、业务逻辑链和现有自动化需求，转换为结构化、可导入、可维护的 `n8n workflow JSON`。与简单的节点拼装不同，该技能强调流程建模、节点职责分离、连接关系清晰、错误路径显式化，以及对生产可用性的基本约束。

## 1. 问题背景

在实际使用 `n8n` 时，很多用户能够描述业务流程，却很难稳定地写出高质量工作流 JSON。常见问题包括：

- 只关注主流程，忽略异常路径
- 过度依赖 `Code` 节点，导致可读性差
- 节点命名随意，难以维护
- Webhook 流程缺少响应闭环
- 没有节流、批处理、去重和重试策略
- 多分支画布混乱，后续接手成本高

因此，一个优秀的自动生成技能不应只是“把需求翻译成 JSON”，而应承担流程架构师的角色。

## 2. 设计目标

该技能围绕五个目标设计：

1. 可导入
   - 输出应尽量符合 `n8n` 工作流导入所需的最小结构。
2. 可读
   - 节点命名、连接关系和画布布局应便于人工理解。
3. 可维护
   - 减少不必要的 `Code` 节点，增强标准节点组合的可追踪性。
4. 可扩展
   - 在用户需求变复杂时，能自然加入审批、重试、通知、等待和批处理。
5. 可优化
   - 不机械照抄用户描述，而是主动提出更好的节点关系和异常处理方式。

## 3. 方法论

技能内部方法不是直接生成 JSON，而是先把用户输入映射成一张流程规划表，核心字段包括：

- 业务目标
- 触发器
- 数据输入
- 目标系统
- 路由规则
- 校验规则
- 成功输出
- 失败处理
- 节流与批处理
- 审批与等待

在此基础上，再把规划结果映射为 `n8n` 节点图，而不是一步到位硬编码 JSON。

## 4. 优化策略

### 4.1 节点选择优先级

技能优先使用 `n8n` 内置节点和原生表达式能力，尽量在以下情况下避免 `Code`：

- 简单字段映射
- 布尔分支
- 多分支路由
- 合并流
- 批量分发

只有在表达式过于复杂、结构化转换明显更适合脚本时，才使用 `Code` 节点。

### 4.2 结构化防错

技能会主动检查并优化这些问题：

- 非触发节点没有上游输入
- 连接关系引用错误
- 节点名重复
- Webhook 没有响应节点
- 外部副作用发生在字段校验之前
- 成功路径和失败路径交织在一起

### 4.3 生产友好性

技能不是纯演示导向，而是带有基础生产意识：

- 有副作用的节点尽量后置
- 对外部 API 预留重试和限流考虑
- 对重复执行场景鼓励设计幂等键
- 对关键流程鼓励添加失败通知

## 5. 支撑资源

为了让技能不仅能“说”，还能“落地”，它配套了三类资源：

- 参考文档
  - 用于规范节点模式、请求模板和交付复核
- 校验脚本
  - 用于快速检查 JSON 是否具备基本可导入结构
- GitHub 友好的文档
  - 方便对外分享、安装和二次维护

## 6. 适用边界

该技能适用于大多数常见业务自动化、Webhook 集成、数据同步、审批流、通知流、AI 增强型流程设计。

它不试图替代以下工作：

- 精确复刻所有第三方节点私有参数
- 为未知私有 API 自动生成绝对正确的请求结构
- 替代真实的生产联调、凭证配置和权限验证

因此，最合适的定位是：
一个面向 `n8n` 自动化设计的高质量生成与优化助手。

## 7. 结论

`n8n-workflow-builder` 的价值，不在于“能不能生成 JSON”，而在于它是否能把用户模糊的业务表达，升级成更清晰、更稳定、更像工程产物的自动化工作流。这个技能的设计重点，是把工作流生成从“文本补全”提升为“结构化自动化建模”。
