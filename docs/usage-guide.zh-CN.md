# n8n Workflow Builder 使用说明

## 1. 技能作用

这个技能的目标，不是只把一句话“翻译成节点”，而是把你的业务思路整理成一个结构更好、可维护性更高、尽量可直接导入的 `n8n workflow JSON`。

它特别适合这几类输入：

- 一串业务步骤
- 一份流程 SOP
- 一个自动化想法
- 一段接口编排逻辑
- 一份已有但混乱的 `n8n` JSON

## 2. 推荐输入方式

你给它的信息越结构化，它生成的结果越稳定。推荐至少包含这几项：

- 业务目标
- 触发方式
- 涉及系统
- 数据输入字段
- 处理步骤
- 分支规则
- 成功结果
- 失败处理

推荐模板：

```text
使用 $n8n-workflow-builder 生成一个完整可导入的 n8n workflow JSON。

业务目标：
- ...

触发方式：
- ...

输入字段：
- ...

执行步骤：
1. ...
2. ...
3. ...

分支规则：
- ...

成功结果：
- ...

失败处理：
- ...
```

## 3. 典型用法

### 从零生成

```text
使用 $n8n-workflow-builder 帮我生成一个 n8n 工作流。

业务目标：
- 当用户提交报名表单后，自动校验信息、写入 Airtable、发送企业微信通知。

触发方式：
- Webhook

输入字段：
- name
- phone
- city

规则：
1. 如果 phone 为空，直接返回失败响应
2. 合法数据写入 Airtable
3. 成功后发送企业微信通知
4. 最后给调用方返回 success
```
### 优化现有工作流

```text
使用 $n8n-workflow-builder 重构下面这个 n8n JSON，目标是减少不必要 Code 节点，优化错误处理，并保持原业务逻辑不变：
{ ...existing workflow json... }
```

### 让它只输出 JSON

```text
使用 $n8n-workflow-builder 根据下面流程生成完整的 n8n workflow JSON，只输出 JSON，不要解释。
```

## 4. 这个技能会默认优化什么

- 触发节点是否合理
- 是否应该先做字段归一化
- 是否应该先校验再执行副作用
- `IF`、`Switch`、`Merge`、`Split In Batches`、`Wait` 是否比 `Code` 更合适
- Webhook 类流程是否有 `Respond to Webhook`
- 是否需要失败通知、重试、节流、去重
- 节点命名是否足够清晰

## 5. 你需要注意什么

- 如果你没有给出真实凭证信息，技能不会伪造 credential ID
- 如果你没有给出精确字段名，技能会基于合理假设生成占位字段
- 如果你的流程会影响生产数据，最好明确说明哪些步骤是测试、哪些是真写入
- 如果你需要完全匹配某个第三方节点版本，最好明确指定

## 6. 生成后如何校验

先把输出保存成 JSON 文件，再运行：

```powershell
python .\scripts\validate_workflow.py .\workflow.json
```

这个脚本会做基础结构检查，但它不是官方 schema 校验器，所以更适合在交付前做快速兜底。

## 7. 最佳实践

- 给出尽量明确的输入字段和分支规则
- 明确哪些动作是同步返回前必须完成的
- 明确失败后是中断、重试还是降级继续
- 对外部 API 调用，说明是否有频率限制
- 对审批流程，说明等待人、超时和通过/拒绝后的路径
